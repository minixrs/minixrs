## 2. Exiting QEMU (`QemuExitCode` + `exit_qemu`)

`src/lib.rs:53-67` â€” writes to the `isa-debug-exit` device's I/O port `0xf4`:

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
#[repr(u32)]
pub enum QemuExitCode {
    Success = 0x10,
    Failed  = 0x11,
}
pub fn exit_qemu(exit_code: QemuExitCode) {
    use x86_64::instructions::port::Port;
    unsafe {
        let mut port = Port::new(0xf4);
        port.write(exit_code as u32);
    }
}
```

Mechanism: the `isa-debug-exit` device causes QEMU to exit with process status `(value &lt;&lt; 1) | 1`. So `Success = 0x10` â†’ host exit code `(0x10&lt;&lt;1)|1 = 33`.

`test-success-exit-code = 33` tells bootimage that host code 33 (from `QemuExitCode::Success`) means the test suite passed; anything else is a failure. `iobase=0xf4,iosize=0x04` matches the `Port::new(0xf4)` / `u32` write.

# Research report: CI smoke/test harness for an aarch64 bare-metal Rust kernel

## 1. Exiting `qemu-system-aarch64` with a controllable exit code from EL1

### Semihosting (recommended, works cleanly for nonzero codes)

**Instruction/register convention (A64, not arm32):** On AArch64 the semihosting trap instruction is `HLT #0xF000` (arm32 uses `SVC 0x123456` in ARM state or `SVC 0xAB` in Thumb; do NOT use those on A64). The call number goes in `W0`/`X0`, the argument (usually a pointer to a parameter block) in `X1`. For exit: `W0 = 0x18` (`SYS_EXIT` / `angel_SWIreason_ReportException`).

**Key finding â€” aarch64 supports nonzero exit codes natively, no `SYS_EXIT_EXTENDED` needed.** This is the important A64-vs-arm32 distinction confirmed in QEMU source (`semihosting/arm-compat-semi.c`):

- On **A64**, `SYS_EXIT`'s argument (`X1`) is a *pointer to a two-word parameter block* `[reason, subcode]`. QEMU logic: if `reason == ADP_Stopped_ApplicationExit (0x20026)`, it exits the host process with `subcode` directly (any value); otherwise it exits with `1`. Source comment: *"The A64 version of SYS_EXIT takes a parameter block, so the application-exit type can return a subcode which is the exit status code from the application."*
- On **A32/T32**, `SYS_EXIT` passes only the reason code in the register (no pointer), so you can only get exit 0 (if `ADP_Stopped_ApplicationExit`) or 1. To return an arbitrary code on arm32 you'd need the optional `SYS_EXIT_EXTENDED (0x20)`. **On aarch64 you do not need `SYS_EXIT_EXTENDED`** â€” the ordinary `SYS_EXIT` extended (pointer) form already carries the code.

So the exit code maps **directly** to the host process exit status (masked to 0â€“255 by POSIX). This is cleaner than blog_os's x86 `isa-debug-exit`, which needs the `(value&lt;&lt;1)|1` remapping (blog_os sets `test-success-exit-code = 33`). On aarch64, `exit(0)` â†’ host 0 (cargo test sees success), `exit(1)` â†’ host 1, etc. No remapping required.

The historical bug where arm semihosting *"always calls exit(0)"* ([Launchpad #1216845](https://bugs.launchpad.net/qemu/+bug/1216845)) was fixed via the semihosting v2 ABI (merged QEMU 4.2). Any modern QEMU (Ubuntu 24.04 ships 8.2.2) is far past this.

**QEMU flags:** Enable with `-semihosting` or `-semihosting-config enable=on,target=native`. The `target` matters for exit-code propagation:
- `target=native` (the default when no debugger, via `target=auto`): QEMU handles the call itself and the exit code reaches the host process. **Use this.**
- `target=gdb`: calls are forwarded to a connected gdb stub â€” you do NOT want this for CI, the host exit status won't reflect the guest.

You can add `chardev`/output redirect, but for exit that's not needed.

**Gotchas:**
- **Security:** QEMU docs are explicit â€” *"Semihosting inherently bypasses any isolation there may be between the guest and the host"* and *"you should only ever run trusted code with semihosting enabled."* Enabling it lets the guest read/write host files, `SYS_READC` can block forever, etc. Irrelevant for trusted CI, but note it.
- **Semihosting only works under TCG, not KVM.** QEMU docs: the trap instructions are reserved and *"most hypervisors trap and fault on them,"* so semihosting is TCG-only. This matters for point 3 below (you can't KVM-accelerate a kernel that exits via semihosting).
- **`-semihosting` changes the meaning of instructions.** With it enabled, `HLT #0xF000` (and on arm32 the semihosting `SVC`/`BKPT` immediates) are interpreted as semihosting calls instead of trapping normally. If your kernel legitimately uses `HLT #0xF000`, behavior changes. In practice fine.
- **Endianness edge case (flag as low-risk):** For a big-endian aarch64 guest on a LE host, `exit(0)` currently comes back as host `1` because the reason word's bytes are swapped and no longer match `0x20026` ([QEMU gitlab #3258](https://gitlab.com/qemu-project/qemu/-/issues/3258)). Does NOT affect a normal little-endian kernel.

### Alternative: PSCI `SYSTEM_OFF` via HVC on `-M virt`

`-M virt` emulates PSCI. From EL1 you load `X0 = 0x84000008` (`PSCI_SYSTEM_OFF`, the 32-bit function ID) and execute `HVC #0` (QEMU virt handles PSCI at EL2/EL3 via HVC/SMC). This is a *clean* shutdown, but **always yields host exit code 0** â€” there's no way to pass a status. So it's fine for a "kernel booted and reached end cleanly" smoke test, but useless for signaling test *failure*. Practical pattern: PSCI `SYSTEM_OFF` on success, and rely on a timeout / semihosting exit(nonzero) / panic for failure. Given you want a controllable code, semihosting is the better primary mechanism.

### Is there an `isa-debug-exit` equivalent on `virt`? â€” No

`isa-debug-exit` (`hw/misc/debugexit.c`, the `(N&lt;&lt;1)|1` device blog_os uses) is an **ISA/port-I/O device, x86-only**. The aarch64 `virt` machine has **no** `isa-debug-exit` and no general `sysbus-debug-exit` equivalent exposed on `virt`. (There is ongoing work on a `virt-ctrl`/`goldfish`-style reset controller in Linux, but nothing you'd rely on for this.) The idiomatic aarch64 answer is semihosting `SYS_EXIT`; that's exactly what the Rust crates below do.

### `-no-reboot` / `-no-shutdown` interplay

- Semihosting `SYS_EXIT` terminates the QEMU process immediately, so `-no-reboot`/`-no-shutdown` don't affect that path.
- They matter for the *failure/crash* path: if a panic/exception triggers a CPU reset loop, `-no-reboot` makes QEMU exit instead of rebooting forever (avoids a hung CI job). `-no-shutdown` does the opposite (keeps QEMU alive after a guest-initiated shutdown) â€” you generally do **not** want `-no-shutdown` in CI. Recommended: add `-no-reboot` as a safety net alongside a workflow-level `timeout`.

## Bottom line / recommended design

- Exit mechanism: **semihosting `SYS_EXIT` (`HLT #0xF000`, `W0=0x18`, `X1`â†’`[0x20026, code]`)** with QEMU `-semihosting-config enable=on,target=native`. ## Deliverable 1 â€” `tests/qemu-boot.expected` (new file, new top-level `tests/` dir â€” safe, the workspace root is virtual so cargo ignores it)

Decision: **11 individual `[as]` substrings, not a count.** A `grep -c` count of 11 cannot say *which* address space is missing and can be satisfied by duplicates; individual name+nr pins give exact failure attribution for free via the checker's report-all loop. Every substring cuts just before the first run-variable field (`ttbr0_pa`, `child_e`, `entry`, `freed`, `at`), keeping the literal-field text (`ttbr0_pa=` etc.) as a stable anchor.

```

# Checked by tools/check-boot-log.sh via `grep -aF` (the log interleaves raw
# single-char tick bytes, so it must be forced to text). Substrings stop just
# before the first run-variable field (ttbr0_pa / child_e / entry / freed / at);
# nr= values are fixed boot-table slots (kernel-shared/src/com.rs).

# Kernel banner (kernel/src/main.rs)
minix.rs booting on aarch64

# Address-space bring-up: 7 servers + init (arch/aarch64/userland.rs)
[as] vm nr=7 ttbr0_pa=
[as] ds nr=5 ttbr0_pa=
[as] vfs nr=1 ttbr0_pa=
[as] sched nr=9 ttbr0_pa=
[as] rs nr=2 ttbr0_pa=
[as] pm nr=0 ttbr0_pa=
[as] init nr=10 ttbr0_pa=

# ...plus the 4 boot stubs (11 [as] lines total)
[as] stub A nr=11 ttbr0_pa=
[as] stub B nr=12 ttbr0_pa=
[as] stub C nr=13 ttbr0_pa=
[as] stub D nr=14 ttbr0_pa=

[ksys SYS_EXEC] target=15 name=worker entry=
[ksys SYS_EXIT] target=w nr=15 freed=
```

## Deliverable 3 â€” `tools/check-boot-log.sh` (new, executable, style-matched to `qemu-run.sh`)

Per-line `grep -aqF` loop (not `grep -f`, whose any-match semantics silently pass when only one expected line matches); reports **all** violations before the non-zero exit. Uses `if â€¦; then continue; fi` rather than `[[ â€¦ ]] &amp;&amp; continue` (the latter trips `set -e` when the guard is false on the last loop line), and `checked=$((checked + 1))` rather than `((checked++))` (which returns status 1 at 0).

```bash
#!/usr/bin/env bash
#
# tools/check-boot-log.sh -- verify a QEMU serial boot log against the
# checked-in marker lists.
#
#   usage: tools/check-boot-log.sh &lt;qemu-log&gt;
#
# Every substring in tests/qemu-boot.expected must appear in the log; no
# substring in tests/qemu-boot.forbidden may appear. Blank lines and '#'
# comment lines in either file are ignored. Every violation is reported (not
# just the first) before the non-zero exit, so one run shows the whole story.
#
# grep needs -a (the log interleaves raw single-char tick bytes, so it reads
# as binary) and -F (the marker files hold literal substrings, not regexes).

set -euo pipefail

if [[ $# -ne 1 ]]; then
    echo "usage: $0 &lt;qemu-log&gt;" &gt;&amp;2
    exit 64
fi

LOG="$1"
ROOT="$(git rev-parse --show-toplevel)"
EXPECTED="$ROOT/tests/qemu-boot.expected"
FORBIDDEN="$ROOT/tests/qemu-boot.forbidden"

if [[ ! -f "$LOG" ]]; then
    echo "error: log file not found: $LOG" &gt;&amp;2
    exit 66
fi

fail=0
checked=0

# Required markers: report every one that is missing.
while IFS= read -r line; do
    if [[ -z "$line" || "$line" == \#* ]]; then continue; fi
    checked=$((checked + 1))
    if ! grep -aqF -- "$line" "$LOG"; then
        echo "MISSING   $line"
        fail=1
    fi
done &lt; "$EXPECTED"

# Forbidden markers: report every one that is present, with its first hit.
while IFS= read -r line; do
    if [[ -z "$line" || "$line" == \#* ]]; then continue; fi
    checked=$((checked + 1))
    if hit="$(grep -am1 -F -- "$line" "$LOG")"; then
        echo "FORBIDDEN $line"
        echo "    first hit: $hit"
        fail=1
    fi
done &lt; "$FORBIDDEN"

if [[ "$fail" -ne 0 ]]; then
    echo "FAIL: $LOG failed the checks above ($checked markers checked)"
    exit 1
fi
echo "PASS: $LOG passed all $checked marker checks"
```

`chmod +x tools/check-boot-log.sh` before committing (matches `qemu-run.sh`).

## Deliverable 4 â€” CI job in `.github/workflows/ci.yml`

Insert after the `miri` job (line 98), before `coverage`, keeping the advisory jobs adjacent. All action SHAs are the ones already pinned in this file.

**Timeout decision: `timeout 45`.** The tracker suggested 25 and the 4.8 verification used 30 locally â€” where 6 alarm fires in 30 s means the first fire (the only timing-sensitive assertion) lands ~5 s after boot on an M-series TCG host. The CI arm64 runner is also native-ISA TCG but on shared, likely slower silicon, and edk2 firmware boot under TCG can eat 10+ s before the kernel banner. 45 s gives roughly 4â€“8x margin over the local first-fire point at a cost of ~15 extra CI seconds; every other assertion is pure boot chatter. Tighten later once real runs are observed.

**Run-step exit handling:** the kernel never exits (WFE idle), so `timeout`'s TERM is the expected terminator and exit 124 the expected status on every good run â€” the step must not fail on it; the checker step is the sole verdict. `cargo kernel-aarch64` runs first as its own step so a compile failure fails *there*, not silently inside the tolerated run step (the `cargo run --release` build is then a cache no-op â€” same profile/target).

```yaml

    name: QEMU boot smoke (advisory)
    # Free arm64 runner for public repos; native-ISA TCG (no KVM requested).
    runs-on: ubuntu-24.04-arm
    # Advisory while it proves stable across a few PRs (phase-5-prep chunk 1);
    # flip to blocking once trusted. Shared-runner TCG timing is the flake risk.
    continue-on-error: true
    timeout-minutes: 20
    steps:
      - uses: actions/checkout@df4cb1c069e1874edd31b4311f1884172cec0e10 # v6.0.3
      - name: Install pinned toolchain
        run: rustup show
      - uses: Swatinem/rust-cache@e18b497796c12c097a38f9edb9d0641fb99eee32 # v2
      - name: Install QEMU + aarch64 UEFI firmware
        run: |
          sudo apt-get update
          sudo apt-get install -y --no-install-recommends qemu-system-arm qemu-efi-aarch64
          qemu-system-aarch64 --version
      - name: Fetch Limine bootloader (pinned commit)
        run: make -C external/limine
      - name: Build kernel
        run: cargo kernel-aarch64
      - name: Boot in QEMU (45 s wall clock)
        env:
          QEMU_EFI_AARCH64: /usr/share/qemu-efi-aarch64/QEMU_EFI.fd
        run: |
          # The kernel runs forever (WFE idle); timeout(1) is the terminator,
          # so exit 124 is the *expected* status of a healthy run. The checker
          # step below is the pass/fail verdict.
          status=0
          timeout 45 cargo run -p minixrs-kernel --target aarch64-unknown-none --release \
            &gt; qemu.log 2&gt;&amp;1 || status=$?
          echo "qemu run finished with status $status (124 = timeout, expected)"
      - name: Check boot log
        run: tools/check-boot-log.sh qemu.log
      - name: Upload boot log
        uses: actions/upload-artifact@ea165f8d65b6e75b540449e92b4886f43607fa02 # v4
        if: always()
        with:
          name: qemu-boot-log
          path: qemu.log
          if-no-files-found: error
```

Artifact upload is `if: always()` (not failure-only): a *green* advisory run's log is exactly what's needed to judge flipping the job to blocking, and to tune the 45 s window. `permissions: contents: read` suffices (coverage already uploads under it).

## Risks / edge cases addressed

- **CI TCG timing:** only first occurrences asserted (`[alarm 1]`, first fork/exec/exit, one D chain); never counts. 45 s window with 4â€“8x margin.
- **`[ksys SYS_EXIT] target=D nr=14` head-carve race** (the one residual timing assertion): `EXIT_TRACE_HEAD = 6`, so D's exit must be among the first 6 exits. Boot interleaving makes this near-certain (stub D faults in its first quantum; a worker cycle needs several SENDRECs + an image load), but if it ever flakes in CI, drop that single line â€” `[pf] proc=D` (unsampled) and the `SYS_KILL â€¦ sig=11` line (kills are rare, always head-carved) already pin the chain.
- **`grep -f` any-match trap:** avoided by design â€” per-line `grep -aqF` loop asserts all.
- **Buffering:** QEMU writes the chardev unbuffered; redirect-to-file is the CLAUDE.md-sanctioned workflow. `[ksys]`/`[pf]` lines are written with IRQs masked, so tick bytes can't split them mid-line.
- **Limine fetch needs network:** isolated in its own CI step so a GitHub/network failure reads as "Fetch Limine" red, not a mystery boot failure; the commit is pinned in `external/limine/Makefile`. Optional later improvement: `actions/cache` on `external/limine/dist` keyed by `LIMINE_COMMIT`.
- **First-run build time:** `timeout-minutes: 20` covers a cold cross-build of kernel + nested server/init/worker builds on the 4-core arm runner; `rust-cache` warms subsequent runs.
- **Exit 124:** run step tolerates any status and just logs it; a QEMU-won't-start failure (e.g. firmware missing, runner exit 69) surfaces as an empty log â†’ checker fails listing every marker, artifact shows why.
